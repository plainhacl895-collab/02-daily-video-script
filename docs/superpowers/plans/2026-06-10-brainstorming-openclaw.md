# 头脑风暴：从想法到设计

通过自然协作对话，将想法转化为完整的设计和规格说明。

首先了解当前项目背景，然后一次一个问题地完善想法。一旦理解你要构建什么，就呈现设计方案并获取用户批准。

**必须等设计获批后才能动手实现。** 这适用于每一个项目，无论看起来多简单。

## 反模式："这太简单了，不需要设计"

每个项目都要走这个过程。待办事项、单功能工具、配置更改——全都不例外。"简单"项目正是未经审视的假设造成最多浪费的地方。设计可以很短（真正简单的项目几句话即可），但必须呈现并获得批准。

## 清单

逐一完成以下步骤：

1. **探索项目上下文** — 检查文件、文档、最近提交
2. **澄清问题** — 一次一个，理解目的/约束/成功标准
3. **提出 2-3 个方案** — 各有取舍，附推荐
4. **呈现设计** — 分段呈现，每段获取用户批准后才继续
5. **写设计文档** — 保存为 `YYYY-MM-DD-主题-design.md`
6. **规格自查** — 快速检查占位符、矛盾、模糊、范围
7. **用户审阅规格文档** — 用户审阅后再继续
8. **转入实施** — 制定实施计划

## 流程

**理解想法：**

- 先检查当前项目状态（文件、文档、最近提交）
- 在细化问题之前，评估范围：如果请求涉及多个独立子系统，立即指出。不要在需要先分解的项目上浪费问题。
- 如果项目太大无法单一规格文档覆盖，帮助用户分解为子项目：独立模块有哪些、如何关联、按什么顺序构建。然后按正常设计流程对第一个子项目进行头脑风暴。每个子项目走一遍 spec → plan → implementation 周期。
- 对于范围合适的项目，一次一个问题地完善想法
- 可能时优先多选问题，开放式也可以
- 每次只问一个问题——如果话题需要更多探索，拆成多个问题
- 聚焦理解：目的、约束、成功标准

**探索方案：**

- 提出 2-3 个不同方案，各有取舍
- 对话式呈现选项，附推荐和理由
- 推荐方案放在最前面并解释为什么

**呈现设计：**

- 一旦你认为理解了要构建什么，就呈现设计
- 每段按复杂度伸缩：简单的几句话，复杂的 200-300 字
- 每段呈现后确认是否正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 准备好回溯澄清任何不清楚的地方

**为隔离和清晰而设计：**

- 将系统分解为更小的单元，每个单元有明确的目的，通过良好定义的接口通信，可以独立理解和测试
- 对每个单元，应该能回答：它做什么、如何使用、依赖什么？
- 不读内部实现能理解一个单元做什么吗？能改内部实现而不破坏使用者吗？如果不能，边界需要改进。
- 更小、边界清晰的单元也更容易处理——你能更好推理能一次性放入上下文的代码，文件聚焦时编辑也更可靠。当文件变得很大，通常是信号表明它做了太多事。

**在现有代码库中工作：**

- 提案前先探索当前结构。遵循现有模式。
- 当现有代码有问题影响工作（如文件过大、边界不清、职责纠缠），将针对性改进纳入设计——好开发者在工作中改进代码的方式。
- 不要提出无关重构。聚焦服务于当前目标。

## 设计之后

**文档：**

- 将验证过的设计（spec）写入设计文档
- 提交设计文档到版本控制

**规格自查：**
写完规格文档后，用新的眼光审视：

1. **占位符扫描：** 有 "TBD"、"TODO"、不完整章节或模糊需求吗？修复它们。
2. **内部一致性：** 各章节有矛盾吗？架构匹配功能描述吗？
3. **范围检查：** 聚焦到足以单个实施计划吗？还是需要分解？
4. **歧义检查：** 有需求可能被两种方式解读吗？如果有，选一种并明确定下来。

发现就修。不需要再审——修完就继续。

**用户审阅门：**
规格审查通过后，请用户审阅书面规格文档再继续。

等待用户回应。如果需要修改，改完重新跑规格审查。只有用户批准后才继续。

**实施：**

- 制定详细实施计划
- 不要再跳到其他环节。实施计划是下一步。

## 核心原则

- **一次一个问题** — 不要一次抛出多个问题
- **多选优先** — 可能时比开放式更容易回答
- **严格 YAGNI** — 从所有设计中移除不必要的功能
- **探索替代方案** — 确定前总是提出 2-3 个方案
- **增量验证** — 呈现设计，获取批准再继续
- **灵活应变** — 哪里不合理就回溯澄清
