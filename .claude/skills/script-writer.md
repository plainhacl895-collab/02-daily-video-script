# 短视频脚本生成技能 v1.0

## 触发条件

用户说"写脚本"/"写一条XX的脚本"/"帮我写XX"时调用。

## 前置条件

- CLAUDE.md 已加载（人设铁律、违禁词、CTA铁律、知识型内容铁律在 CLAUDE.md 中，技能内不再重复）
- 工作目录为项目根目录

---

## 核心原则：思考，不是执行

**你不是在"选中"一个已有结论当中心思想，而是在综合消化多来源信息后"设计"出一个新的判断。**

MD 分析文件里的结论是素材，不是答案。如果中心思想可以直接在 MD 文件里找到——你没有在思考，你在搬运。

---

## 工作流

### Step 1 — 获取素材 + 策略对齐

```bash
python engine.py status    # 今日策略/支柱/格式推荐
```

读 `memory/layer1-identity/positioning.md` 确认当前定位。

**自问**：今天这条视频发布后，观众会记住"佳佳 = 上海板块分析专家"吗？如果换其他中介来念也没区别 → 换角度。

用户给素材直接进入下一步；没给素材读 `memory/layer3-execution/material-source-guide.md` 按框架引导；引导失败读 `memory/layer3-execution/inspiration-topics.md` 从话题弹药库选题。抖音链接用 `python douyin_extractor.py <链接> --asr --comments` 提取，分析完技法写入 `memory/layer3-execution/competitor-learning-log.md`（不要直接改技能文件）。

小区分析数据可用时（来自 `../03-shanghai-community-analysis/` 或实地探盘），优先做小区深度分析，而非泛话题脚本。一个小区 = 一条视频，系列化输出。

---

### Step 2 — 外部搜索（必做，不可跳过）

**目的**：跳出 MD 文件的框，找到冷数据之外的血肉。

**搜索清单（按脚本类型）**：

| 脚本类型 | 搜索方向 | 搜索平台 | 搜什么 |
|---------|---------|---------|--------|
| 小区分析 | 真实住户评价 | 小红书、安居客问答 | "XX小区 居住体验"、"XX小区 优缺点" |
| 小区分析 | 历史/背景/争议 | 百度、微信公众号 | "XX小区 历史"、"XX小区 化工厂/地块" |
| 小区分析 | 房源实况 | 贝壳/链家当前在售 | 看真实户型照片、装修状态、采光 |
| 板块类 | 第三方分析文章 | 知乎、微信公众号 | "XX板块 分析"、"XX板块 买房" |
| 任何类型 | 竞品内容 | 小红书 | 同话题高赞视频/笔记（看别人选了哪个角度） |
| 故事类 | 人物故事 | 小红书、百度 | 跟主题相关的真实人物经历 |

**输出格式（强制）**：

```text
## 外部搜索简报

**平台**：<搜索来源>
**搜索词**：<实际使用的搜索词>
**发现**：
- <具体引用1，含出处链接>
- <具体引用2，含出处链接>
**脚本使用计划**：<这一条用在哪个段落，为什么>
```

**铁律**：
- 每个脚本至少搜 2 个不同平台
- 至少找到 2 条 MD 文件中没有的、可用于脚本的信息
- 如果搜完发现没有任何新信息可补充，在搜索简报中声明 "无增量信息" 并说明原因，然后进入下一步

---

### Step 3 — 数据洞察（小区分析类必做）

只针对 pillar=Property 且 strategy=专业分析，其他类型跳过。

1. 从 `../03-shanghai-community-analysis/` 读文件：
   - `小区分析/{区域}/{小区名}_深度分析.md`
   - `板块分析/{板块名}_深度分析.md`
   - `小区对比总表.csv`
2. **查实时房源**：`python engine.py listings <小区名>`
3. 从 CSV 提取同板块 2-3 个直接竞品，5 维度并排对比：挂牌均价、成交周期、建成年份、物业费/评级、车位配比
4. 找出反直觉的数字关系（至少 2 个）
5. 输出 **数据洞察简报**：

```text
## 数据洞察

**对比小区**：<小区名> vs <竞品1> vs <竞品2>
**核心发现**：<一句话——反直觉的现象>
**支撑数据**：
| 维度 | <目标小区> | <竞品1> | <竞品2> |
|------|-----------|---------|---------|
| 均价 | X万 | Y万 | Z万 |
| 成交周期 | X天 | Y天 | Z天 |
| 物业 | X元/X星 | Y元/Y星 | Z元/Z星 |
| 车位 | 1:X | 1:Y | 1:Z |
| 建成年份 | XXXX | XXXX | XXXX |
**推理链**：<这个数据背后的逻辑——不是"A比B贵"，是"A贵因为X，B便宜但代价是Y">
**适合人群重定义**：<不能用泛泛标签，必须基于数据给出精确画像>
**数字验收**：<列出本脚本会用到的每一个硬数字及其来源——CSV/链家/内网>
```

**铁律**：每一个硬数字必须能在源文件中找到出处。拿不准的标 `[待确认]`，宁可不写不能编。

---

### Step 4 — 素材提炼 + 准入检查

综合 MD 文件 + 外部搜索结果 + 数据洞察，设计一条**不存在于任何单一来源中的新判断**作为中心思想。

输出格式：

```text
## 素材提炼结果

**中心思想**：<一句话可争论的判断>
**信息密度**：数字(有/无) / 冲突(有/无) / 决策点(有/无) / 反转(有/无)
**定位契合度**：高/中/低 — <理由>
**人设安全性**：有利 / 中性 / 不利 — <判断>
**画面可行性**：可实拍 / 可混剪 / 仅口播 — <理由>
**实物锚点**：<绑定的具体事物，无则标"无">
```

#### 中心思想张力审查（三项至少满足两项）

| 维度 | 检查方式 | 反面案例 |
|------|---------|---------|
| 可争论 | 是否有人能合理反对这个判断？ | "物业很重要" — 没人反对，正确的废话 |
| 反直觉 | 是否跟大多数人的第一反应相反？ | "买房子要看地段" — 不用你说 |
| 具体 | 换一个小区/板块还能讲吗？能讲就是太泛 | "天山板块配套成熟" — 换个板块名也能讲 |

不通过 → 重新设计中心思想。通过 → 继续。

准入标准（五项至少满足四项）：可争论性、有具体信息、定位相关、可拍摄、有实物锚点。人设安全性"不利"一票否决。

**→ 提交审批，等用户说"可以"/"继续"/"写吧"。**

---

### Step 5 — 叙事结构设计

**不要机械套用模板**。根据中心思想设计段落结构的推进逻辑：

| 段落 | 叙事功能 | 情绪/认知推进 |
|------|---------|-------------|
| 开场 | <这段在叙事里承担什么角色> | <观众看完这段的感受/想法> |
| 段落1 | <论证中心思想的哪个侧面> | <认知变化> |
| 段落2 | <论证中心思想的哪个侧面> | <认知变化> |
| ... | ... | ... |
| 收束 | <回扣中心思想> | <留下的记忆点> |

**自检清单**：
- 每一段有明确的叙事功能，不是"模板要求我写这段所以我写"
- 段落之间有逻辑推进关系（不是并列堆砌）
- 缺点段落是真实信息的输出，不是"因为我被要求说缺点"
- 如果有两段在讲同一个意思 → 合并或删掉一段

---

### Step 6 — 写脚本

写 YAML frontmatter + 正文的 `.txt` 文件，保存到 `$env:USERPROFILE\Desktop\daily-script-YYYY-MM-DD.txt`。

```text
---
date: YYYY-MM-DD
format: talking_head | property_walk | mixed_montage
pillar: Market | Property | Story
strategy: 陪伴决策 | 专业分析 | 避坑指南 | 资源推荐
bgm: <风格描述>
scene: <拍摄场景>
shot: <机位描述>
word_count_target: <字数范围>
best_platform: <平台>
---

# 标题

## 开场（3秒钩子）
[画面:xxx] [语速:正常]

## 主体段落
每段只论证中心思想的一个侧面
[画面:xxx] [字幕叠加:xxx] [语速:xxx]

## 金句
从金句库选或即兴写
[画面:xxx] [语速:慢]

## 结尾
引导点赞/关注，不做成交暗示
[画面:xxx] [字幕叠加:关注引导]
```

每段必须标注 `[画面]` `[字幕叠加]` `[语速]`。字数按内容需要决定，不设上限——2分钟、2分半都可以，内容够就不缩短。

写前读 `memory/layer3-execution/copywriting-techniques.md` 自查清单。

**小区分析系列额外要求**：
- 关键数字必须叠加字幕，不靠观众耳朵记
- 不在孤立讲一个小区，放在板块内对标
- 有实拍就不用网图
- 每期结尾预告下一期

### Step 7 — 验证 + 自查

```bash
python viral_validator.py <脚本路径>    # 9维爆款验证（27分制）
python pleasure_scorer.py <脚本路径>    # 爽点评分（12分制）
```

验证器是机械评分。中心思想明确性、字数超标等项如果人工审计通过，忽略机械低分。

**人设自查**：
- 违禁词扫一遍（在 CLAUDE.md 中定义）
- 语气是朋友聊天不是上课/推销
- "smell test"：1000万预算客户读到每句话，是觉得"帮我分析"还是"吓我"？

---

### Step 8 — 记录 + 输出制作指南

```bash
# 记录生成
python -c "from engine import VideoEngine; e = VideoEngine(); e.record_generation({'strategy':'专业分析','pillar':'Property','video_format':'property_walk','topic':'<标题>','word_count':350,'duration_sec':90,'pleasure_score':8,'community':'<小区名>','district':'<板块>','series':'<系列名>','episode':1})"
```

> ⚠ `record_generation` 按日期 upsert——同一天多条脚本，后一条覆盖前一条。保留多版本需先备份 `data/history.json`。
> ⚠ `history.json` 的 `max_window: 30` 会丢弃超过 30 条的旧记录。长期趋势分析需在 25 条左右时手工备份或调大 `max_window`。

制作运营建议和拍摄清单参考 `memory/layer3-execution/production-playbook.md` 和 `solo-shooting-guide.md` 输出。

---

### Step 9 — 反馈录入（发布后）

用户发布后回填播放数据：

```bash
python -c "from engine import VideoEngine; e = VideoEngine(); e.record_feedback('YYYY-MM-DD', {'views':1200,'completion_rate':0.45,'engagement_rate':0.038,'likes':56,'comments':12,'shares':8})"
```

有客户通过视频来咨询 → 记录到 `memory/layer4-feedback/client-inquiry-log.md`。积累 ≥3 条已发布数据后跑 `python engine.py analyze`。
